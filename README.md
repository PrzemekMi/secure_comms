# secure_comms

A Python cryptographic communications system for two-party secure messaging over a mock [Meshtastic](https://meshtastic.org/) LoRa mesh channel. Trust is anchored entirely to physical YubiKey PIV hardware — no network CA, no third-party certificates.

> Designed for high-assurance, air-gapped or LoRa-only mesh environments where traditional PKI is unavailable.

## Project Architecture

secure_comms simulates a bilateral secure session between **Side A** (initiator) and **Side B** (responder). The only shared state established before runtime is each party's long-term RSA public key, exchanged once in person during a key ceremony.

### Module Overview

| Module | Role |
|---|---|
| `yubikey/` | `YubiKeyInterface` ABC, `MockYubiKey` (in-memory RSA-2048), `RealYubiKey` stub |
| `ceremony/` | `CeremonyOrchestrator` — one-time in-person bilateral public key exchange with SHA-256 fingerprint verification |
| `channel/` | `MeshtasticChannel` mock — broadcast, fragmentation (7-byte header + 220-byte body), reassembly, configurable latency/loss, 30 s buffer timeout |
| `side_a/` | `SideANode` — waits for `EPHEMERAL_OFFER` from B, encrypts Fernet key under B's ephemeral pub key, signs ciphertext with long-term YubiKey |
| `side_b/` | `SideBNode` — generates fresh RSA-2048 ephemeral key pair per session, signs ephemeral pub key with long-term YubiKey, decrypts Fernet key, **immediately discards ephemeral private key** |
| `demo/` | Interactive CLI + automated smoke test |

### Session Flow

```
Side B                                    Side A
  |                                          |
  |-- generate ephemeral RSA-2048 key pair --|
  |-- sign ephemeral pub key (YubiKey) ----->|
  |                                          |-- verify B's signature
  |                                          |-- generate Fernet key
  |                                          |-- encrypt Fernet key (OAEP-SHA256)
  |                                          |-- sign ciphertext (PSS-SHA256)
  |<-----------------------------------------|
  |-- verify A's signature                   |
  |-- decrypt Fernet key                     |
  |-- discard ephemeral private key          |
  |                                          |
  |<====== encrypted Fernet messages =======>|
```

### Cryptographic Primitives

| Primitive | Usage |
|---|---|
| RSA-2048 OAEP-SHA256 | Fernet key encapsulation |
| RSA-PSS MGF1-SHA256 + MAX_LENGTH salt | Long-term and ephemeral signatures |
| Fernet (AES-128-CBC + HMAC-SHA256) | Symmetric message encryption |
| TTL = 300 s | Applied to all Fernet tokens |

### Forward Secrecy Status

**Implemented — partial.**

Each session uses a fresh RSA-2048 ephemeral key pair generated by Side B. After the handshake, the ephemeral private key is immediately nulled and deleted (`side_b/node.py:196–220`). If B's long-term YubiKey is compromised in the future, past session Fernet keys cannot be recovered because the ephemeral private key no longer exists.

**Known limitations (see AV-005, AV-006):**

- The ephemeral private key and Fernet session key are Python `bytes` / `RSAPrivateKey` objects. Python does not guarantee immediate memory clearing on `del` — the garbage collector may leave key material in RAM for an indeterminate window. Cryptographic zeroing (e.g. `ctypes.memset`) is not yet applied.
- The Fernet session key (`_fernet_key`) persists in the `SideBNode` object for the lifetime of the session. It is not cleared between messages.
- Forward secrecy covers **past sessions only**. The current session's Fernet key is in memory until the node is torn down.

## White-Hat Security Agent

This project uses a persistent **white-hat security researcher** Claude agent that is invoked automatically whenever new features or significant code changes are introduced.

### What the Agent Does

- **Attack vector analysis** — systematically evaluates every new feature against STRIDE, OWASP Top 10, and common crypto attack patterns
- **Threat model maintenance** — adds newly discovered threats to `THREAT_MODEL.md` and marks existing threats as mitigated or re-opened
- **Mitigation recommendations** — provides specific, code-level mitigations prioritised by risk (Critical → High → Medium → Low)
- **Institutional memory** — accumulates security knowledge across sessions in `.claude/agent-memory/white-hat-security-researcher/`

### Review Methodology

1. Understand the feature — data handled, trust boundaries crossed
2. Map the attack surface — all entry points
3. Apply STRIDE across all components
4. Verify against OWASP Top 10
5. Audit dependencies for known CVEs
6. Trace sensitive data flows end-to-end
7. Consider chained / compound attack scenarios

## Security Findings

Initial full security review completed **2026-04-15**. 14 attack vectors identified (AV-001 – AV-014).

### Severity Summary

| Severity | Count | IDs |
|---|---|---|
| Critical | 2 | AV-001, AV-002 |
| High | 5 | AV-003 – AV-007 |
| Medium | 4 | AV-008 – AV-012 |
| Low | 2 | AV-013, AV-014 |

### Finding Details

#### Critical

| ID | Finding |
|---|---|
| AV-001 | `session_id` predictability — derived from seeded `random.Random`, not `secrets`/`os.urandom` |
| AV-002 | Handshake lacks replay defense — no nonce binding; Fernet TTL alone is insufficient |

#### High

| ID | Finding |
|---|---|
| AV-003 | `fragment_total` field unbounded — enables memory exhaustion (DoS) |
| AV-004 | PSS `salt_length=MAX_LENGTH` creates verification asymmetry on real hardware |
| AV-005 | Ephemeral private key not cryptographically zeroed — GC-dependent clearance |
| AV-006 | Fernet key survives session in plaintext heap |
| AV-007 | No per-sender cap on reassembly buffers — amplified memory DoS |

#### Medium

| ID | Finding |
|---|---|
| AV-008 | `sig_len` integer not bounds-checked — large allocation before rejection |
| AV-009 | `session_id` counter starts at 0 and wraps predictably |
| AV-010 | `RealYubiKey.sign()` passes digest, not raw data — padding oracle risk when wired |
| AV-011 | `channel._rng` seeded publicly in demo — breaks packet loss randomness |
| AV-012 | `demo` status command exposes internal state via direct attribute access |

#### Low

| ID | Finding |
|---|---|
| AV-013 | No timeout on `EPHEMERAL_OFFER` — stale ephemeral key reuse window |
| AV-014 | Fernet uses AES-128, not AES-256 |

### Recurring Patterns to Watch

- **RNG** — `channel` uses `random.Random`; acceptable for simulation, dangerous if promoted to production session ID generation
- **Memory** — `_fernet_key` and `_eph_private_key` are Python `bytes` objects and are not zeroed on deletion
- **Packet parsing** — all length fields in packet headers must be bounds-checked against `MAX` before buffer allocation
- **Threat model gap** — handshake replay (THREAT-2) is marked *Mitigated* but lacks nonce binding; session hijacking has no threat entry
